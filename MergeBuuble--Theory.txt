1. The Headers (The Toolkits)
C++
#include <iostream>
#include <vector>
#include <omp.h>

using namespace std;
#include <iostream>: Imports the standard input/output stream. This gives you cout (to print to the screen) and cin (to read keyboard input).

#include <vector>: Imports the C++ Vector library. A vector is a "smart array" that can automatically resize itself. We use this instead of standard arrays int arr[10] because we don't know how big the array will be until the user types n.

#include <omp.h>: The most important line. This imports the OpenMP library, giving you access to functions like omp_get_wtime() to measure execution time.

#include <cstdlib>: Imports the C Standard General Utilities Library. We need this specifically for the rand() function, which generates the random numbers for our massive performance test.

using namespace std;: Tells the compiler we are using the "standard" namespace, so we don't have to type std::cout and std::vector every single time.

2. Sequential Bubble Sort (seqBubble)
C++
void seqBubble(vector<int>& arr) {
    int n = arr.size();
    for (int i = 0; i < n - 1; i++) {
        for (int j = 0; j < n - i - 1; j++) {
            if (arr[j] > arr[j + 1]) swap(arr[j], arr[j + 1]);
        }
    }
}
vector<int>& arr: The & is crucial. It means "pass by reference." We are passing the actual array into the function, not a copy. If we didn't use &, the function would sort a copy of the array and throw it away, leaving our original array unsorted!

int n = arr.size();: Grabs the length of the array.

n - i - 1: A classic Bubble Sort optimization. After the first loop (i=0), the largest number is guaranteed to be at the very end. We don't need to check the last number again, so we subtract i to make the loop shorter every time.

swap(arr[j], arr[j + 1]): A built-in C++ function that swaps two values if the left one is bigger than the right one.

3. The Helper Function: merge
C++
void merge(vector<int>& arr, int left, int mid, int right) {
    int n1 = mid - left + 1;
    int n2 = right - mid;
    vector<int> L(n1), R(n2);
Why was this created? Merge Sort works by splitting an array into tiny pieces, sorting them, and then "merging" them back together. This function does the merging. Both Sequential and Parallel Merge Sort use this exact same helper function.

int n1 and int n2: Calculates the sizes of the Left half and Right half of the split array.

vector<int> L(n1), R(n2);: Creates two temporary arrays (Left and Right) to hold the split data.

C++
    for (int i = 0; i < n1; i++) L[i] = arr[left + i];
    for (int j = 0; j < n2; j++) R[j] = arr[mid + 1 + j];
These loops simply copy the actual numbers from the main arr into our temporary L and R arrays so we can compare them.

C++
    int i = 0, j = 0, k = left;
    while (i < n1 && j < n2) {
        if (L[i] <= R[j]) arr[k++] = L[i++];
        else arr[k++] = R[j++];
    }
This is the core logic of merging. It points a finger at the start of L (i) and R (j). It asks: "Which number is smaller?" It takes the smaller number, overwrites the main arr with it, and moves the finger forward (++).

C++
    while (i < n1) arr[k++] = L[i++];
    while (j < n2) arr[k++] = R[j++];
}
Why are these here? If the Left array runs out of numbers before the Right array (or vice versa), these loops just dump the remaining leftover numbers into the main array.

4. Sequential Merge Sort (seqMerge)
C++
void seqMerge(vector<int>& arr, int left, int right) {
    if (left < right) {
        int mid = left + (right - left) / 2;
        seqMerge(arr, left, mid);
        seqMerge(arr, mid + 1, right);
        merge(arr, left, mid, right);
    }
}
if (left < right): This is the base case for recursion. It means "keep splitting the array until the pieces are only 1 element long."

left + (right - left) / 2: Finds the middle index. We do it this way instead of (left + right) / 2 to prevent "Integer Overflow" if the array is billions of elements long.

seqMerge(...): It calls itself (recursion) to continuously break the left half, then the right half, down into smaller pieces.

merge(...): Once broken down, it calls the helper function to stitch them back together in sorted order.

5. Parallel Bubble Sort (parBubble) - Highly Questionable in Viva
C++
void parBubble(vector<int>& arr) {
    int n = arr.size();
    for (int phase = 0; phase < n; phase++) {
Why "phases"? We cannot parallelize a normal Bubble Sort because threads will try to swap the same numbers at the same time, crashing the program. We use Odd-Even Transposition Sort. It takes exactly n phases to sort an array this way.

C++
        if (phase % 2 == 0) { // Even Phase
            #pragma omp parallel for
            for (int i = 0; i < n - 1; i += 2) {
                if (arr[i] > arr[i + 1]) swap(arr[i], arr[i + 1]);
            }
if (phase % 2 == 0): This checks if we are on an "Even" phase (0, 2, 4...).

#pragma omp parallel for: Tells OpenMP to divide the loop among all CPU cores.

i += 2: CRITICAL! Notice we jump by 2, not 1! This means Thread A looks at indexes (0,1), Thread B looks at (2,3), Thread C looks at (4,5). Because they don't overlap, there is no race condition! We don't even need #pragma omp critical.

C++
        } else { // Odd Phase
            #pragma omp parallel for
            for (int i = 1; i < n - 1; i += 2) {
                if (arr[i] > arr[i + 1]) swap(arr[i], arr[i + 1]);
            }
        }
    }
}
int i = 1: On Odd phases, we shift our focus. Thread A looks at (1,2), Thread B looks at (3,4). By alternating Even and Odd phases, the large numbers "bubble" to the top safely.

6. Parallel Merge Sort (parMerge)
C++
void parMerge(vector<int>& arr, int left, int right) {
    if (left < right) {
        int mid = left + (right - left) / 2;
        
        #pragma omp task shared(arr)
        parMerge(arr, left, mid);
        
        #pragma omp task shared(arr)
        parMerge(arr, mid + 1, right);
        
        #pragma omp taskwait
        merge(arr, left, mid, right);
    }
}
#pragma omp task: Instead of executing the left half immediately, this tells OpenMP: "Hey, package this recursive function call as a 'Task' and throw it in a pool." Any idle CPU thread will grab it and start sorting the left half. The second task does the same for the right half. This is how we get massive parallelism.

shared(arr): Tells the tasks that they should all be editing the exact same master array in memory, not creating their own private copies.

#pragma omp taskwait: CRITICAL! This forces the current thread to pause. It says: "Do not execute the merge function until both the Left Task and Right Task are 100% finished." Without this, it would try to merge unsorted data!

7. Main Execution & Performance Testing
Why do we separate logic testing and performance testing?
If you try to measure the speed of parallel vs sequential using 10 elements, Sequential will win. OpenMP takes a tiny fraction of a second to "wake up" the CPU threads. For 10 elements, the sorting is done before the threads even wake up!
To show actual performance, we must give the CPU something heavy to chew on (like 20,000 elements).

C++
    int bigN = 20000;
    vector<int> a(bigN), b(bigN), c(bigN), d(bigN);
bigN = 20000: This is the heavy load. If the examiner asks to calculate "actual performance with more load," you simply change this to 50000 or 100000.

Why 4 arrays (a, b, c, d)? If we sorted array a with seqBubble, and then passed array a to parBubble, parBubble would be sorting an already sorted array. It would finish in 0.00 seconds! We create 4 identical copies so each algorithm gets a fair fight on completely unsorted data.

C++
    start = omp_get_wtime();
    seqBubble(a);
    end = omp_get_wtime();
omp_get_wtime(): Stands for "Wall Time". It captures the exact machine time before the function starts, and right after it finishes. Subtracting them gives you the exact benchmark time.

C++
    #pragma omp parallel
    {
        #pragma omp single
        parMerge(d, 0, bigN - 1);
    }
Why is parMerge wrapped in this block? #pragma omp parallel wakes up the team of threads.
If we didn't use #pragma omp single, every single thread would try to start sorting the entire array from scratch. single ensures that only one "Master" thread triggers the initial root division of the array. Once it creates the first two Tasks, the other sleeping threads wake up and grab those tasks.

Part 1: The Basics of Sorting (Time & Space Complexity)

1. What is the fundamental difference between Bubble Sort and Merge Sort?
Bubble Sort is a simple comparison-based algorithm that swaps adjacent elements to "bubble" the largest element to the end. Merge Sort is a Divide-and-Conquer algorithm that splits the array into single elements and merges them back together in sorted order.

2. What is the Time Complexity (TC) of Sequential Bubble Sort?
Worst/Average Case: $O(N^2)$ (When the array is reverse sorted or jumbled).Best Case: $O(N)$ (If the array is already sorted, and we use a flag to break the loop early).

3. What is the Space Complexity (SC) of Sequential Bubble Sort?
$O(1)$. It is an "in-place" sorting algorithm, meaning it doesn't need any extra memory arrays; it just swaps elements within the original array.

4. What is the Time Complexity (TC) of Sequential Merge Sort?
It is strictly $O(N \log N)$ in the Best, Average, and Worst cases. It will always divide the array $\log N$ times and take $N$ steps to merge them.

5. What is the Space Complexity (SC) of Merge Sort?
$O(N)$. It is NOT an in-place sort. It requires temporary Left and Right arrays (or a main temporary array) to hold the data while merging.

6. Which sorting algorithm is better and why?
Merge Sort is drastically better for large datasets because $O(N \log N)$ scales much better than $O(N^2)$. However, for very small arrays (e.g., 5-10 elements), Bubble Sort can be faster because it doesn't have the overhead of recursive function calls.

7. Are Bubble Sort and Merge Sort "Stable" algorithms?
Yes, both are stable. "Stable" means that if there are two identical numbers in the array, their relative order remains the same after sorting.Part 2: OpenMP & Parallel Computing Foundation

8. What is OpenMP?
OpenMP (Open Multi-Processing) is an API used for shared-memory parallel programming in C, C++, and Fortran.

9. What memory architecture does OpenMP use?
It uses a Shared Memory architecture. All threads have access to the same global memory (the RAM of the single computer you are running it on).

10. What is the Fork-Join Execution Model?
An OpenMP program starts as a single "Master" thread. When it hits a #pragma omp parallel block, it "Forks" (creates) a team of worker threads. They execute in parallel. At the end of the block, they "Join" (synchronize and terminate), and only the master thread continues.

11. How do you compile an OpenMP program in Ubuntu?
By using the -fopenmp flag. Example: g++ -fopenmp file.cpp -o file.

12. What happens if you compile OpenMP code without the -fopenmp flag?
The compiler simply ignores the pragmas (treating them as comments) and runs the program as a standard, single-thread sequential program without giving any errors.

13. What is a Thread?
A thread is the smallest unit of execution that an OS can schedule. Multiple threads run inside a single process and share the same memory space.

14. What is a Race Condition?
It happens when multiple threads try to read and write to the exact same memory location at the exact same time, causing data corruption.Part 3: Parallel Bubble Sort (Odd-Even Transposition)

15. Can you directly parallelize standard Bubble Sort?
No. Standard Bubble Sort has a strict loop dependency. You cannot swap index 1 and 2 if another thread is currently trying to swap index 2 and 3. It will corrupt the array.  

16. So how did you parallelize Bubble Sort?
I used a variation called the Odd-Even Transposition Sort. It separates the array into independent pairs that don't overlap.  

17. How does Odd-Even Transposition Sort work?It runs in two alternating phases:Even Phase: Compares and swaps Even indexes with their right neighbor: (0,1), (2,3), (4,5).Odd Phase: Compares and swaps Odd indexes with their right neighbor: (1,2), (3,4), (5,6).

18. Why does Odd-Even sorting prevent race conditions?
Because in any given phase, no two threads are looking at the same index. For example, in the Even phase, Thread A locks indexes 0 and 1, and Thread B locks indexes 2 and 3. There is no overlap, so no #pragma omp critical is needed!   

19. How many phases does Odd-Even Sort take to guarantee the array is sorted?
It requires exactly $N$ phases (where N is the number of elements in the array).  

20. What is the Time Complexity of Parallel Odd-Even Sort?
If you have $N$ processors, the time complexity drops from $O(N^2)$ to $O(N)$.  

21. Is Parallel Bubble Sort actually used in real-world High-Performance Computing?
No. Even when parallelized, $O(N)$ or $O(N^2/P)$ is still horribly slow compared to parallelizing $O(N \log N)$ algorithms. It is mostly taught as a beginner concept for learning loop parallelization.  

22. Explain #pragma omp parallel for used in Bubble Sort.
It tells OpenMP to take the iterations of the for loop and divide them equally among the available CPU threads.Part 4: Parallel Merge Sort (Tasks & Divide/Conquer)

23. How did you parallelize Merge Sort?
I used OpenMP Tasks (#pragma omp task) to parallelize the recursive divide-and-conquer calls.

24. Why didn't you use #pragma omp parallel for for Merge Sort?
Because omp for requires a standard loop with a known number of iterations. Merge Sort uses Recursion. You cannot use omp for on a recursive function.

25. What does #pragma omp task do?
When a thread hits this line, instead of executing the function immediately, it packages the function call into a "Task" and throws it into a shared task pool. Any idle thread can grab a task from the pool and execute it.

26. Why do we need #pragma omp taskwait in Merge Sort?
It acts as a barrier. It tells the thread: "Do not move to the next line of code until all the child tasks you spawned are 100% complete."

27. What happens if you remove taskwait?
The code would try to run the merge() function before the left and right halves are actually finished sorting. The array would remain a jumbled mess.

28. Why do we wrap the first Merge Sort call in #pragma omp single?
If we just used #pragma omp parallel, every single thread in the CPU would simultaneously try to start sorting the entire array from index 0. #pragma omp single ensures only one Master Thread starts the root task. Once that master thread splits the array into two tasks, the other sleeping threads wake up and grab them.

29. What is the theoretical Time Complexity of Parallel Merge Sort?
$O(N \log N / P)$, where P is the number of processors.

30. Is the actual merge() logic parallelized in your code?
No, in the standard implementation, the dividing is parallelized, but the stitching back together (merge) is done sequentially by whoever finished the task. (Note: Parallelizing the merge step is possible but highly complex).Part 5: Performance, Benchmarking, and OpenMP Concepts

31. How did you measure performance in your code?
I used omp_get_wtime(), which returns the current wall-clock time in seconds. I recorded the time before calling the sort function and after, then subtracted the two.

32. What is "Speedup"?
Speedup = (Execution time of the Sequential algorithm) / (Execution time of the Parallel algorithm). 

 33. What is "Efficiency"?
Efficiency = Speedup / Number of Processors. It measures how well the hardware is being utilized. 

 34. Why did your code generate a massive array (e.g., 20,000 elements) for performance testing?
Because creating threads, distributing tasks, and destroying threads takes a few microseconds. If we sort only 10 elements, the Sequential sort finishes faster than the OS can even create the parallel threads! We need a huge array so the sorting work outweighs the thread creation overhead.

35. Why did you create 4 separate arrays (a, b, c, d) with identical random numbers?
To make the benchmark fair. If I used the same array for all four tests, the first function would sort it. Then the next three functions would be sorting an already sorted array, making their times falsely look like 0.00 seconds.

36. What is Amdahl’s Law?
It states that the maximum speedup of a parallel program is strictly limited by the portion of the code that must remain sequential. (e.g., You can parallelize the dividing in Merge Sort, but the final big merge is sequential, capping your max speed).

37. What is omp_get_max_threads()?
It returns the maximum number of logical processors/threads your hardware can support.Part 6: Scenario-Based & "Trick" Questions (External Examiner Favorites)

47. Examiner: "What is a barrier in OpenMP?
"Answer: "A barrier is a synchronization point where all threads must stop and wait until every single thread in the team has reached that point. #pragma omp for has an implicit barrier at the end unless you use nowait."

"Answer: "No, taskwait only forces the specific thread that spawned the child tasks to wait. Other threads in the background are still free to execute those tasks."
