1. Standard C++ and Data Structures
#include <vector> and vector: A vector is a dynamic array in C++. We use it instead of standard arrays because it automatically resizes itself. It is essential for representing the graph and storing queues of varying sizes.

class Graph: This is an Object-Oriented concept. It bundles our graph's data (the number of vertices and the edges) and its behaviors (the BFS and DFS functions) into one neat package.

adj (Adjacency List): Declared as vector<vector<int>> adj. This is a list of lists used to represent the graph. If adj[0] contains 1 and 2, it means Node 0 is directly connected to Node 1 and Node 2.

push_back(value): A C++ vector function that adds a new element to the end of a vector. We use it to build the graph and to add nodes to our queues.

insert(end, start_of_new, end_of_new): A C++ vector function used in the BFS code. Instead of adding elements one by one, it grabs an entire chunk of elements from one vector and pastes them at the end of another.

visited array: Declared as vector<bool> visited. This acts as a checklist. Before we process a node, we check if it is false (unvisited). Once processed, we mark it true so we do not process it again and get stuck in an infinite loop.

2. OpenMP Parallelization Pragmas (The Core HPC Concepts)
#pragma omp parallel: This is the master switch for parallelization. Whenever the code hits this line, it creates a "team" of threads. Any block of code directly under this will be executed by all threads in the team simultaneously.

#pragma omp for nowait: Used in BFS. The for tells OpenMP to divide the iterations of the loop equally among the available threads. The nowait keyword is a crucial optimization: it tells threads that finish their chunk of the loop early that they do not have to wait around for the slower threads to finish; they can immediately move on to the next task.

#pragma omp critical: The most important safety guard in parallel programming. It creates a "bottleneck" where only one thread is allowed to enter at a time. We use this whenever threads are writing to shared variables (like printing to cout, updating the visited array, or merging vectors). Without this, threads would overwrite each other's data, causing crashes.

#pragma omp single: Used in DFS. It tells the OpenMP team: "Only let one thread execute the following block." We use this to trigger the very first root node in our recursive DFS. Once that first thread starts traversing, it will spawn other tasks.

#pragma omp task: Used in DFS. Instead of forcing threads to execute immediately, this packages a function call (like dfsRecursive) into a "task" and throws it into a pool. Whenever a CPU thread is free, it grabs a task from the pool and executes it. This is the only safe way to parallelize recursive functions.

3. OpenMP Utility Functions
omp_get_wtime(): Stands for "wall time." It returns the current time in seconds. By calling this before and after our algorithm and subtracting the difference, we get the exact execution time to measure our performance.

omp_get_max_threads(): Returns the maximum number of CPU threads your specific computer can run simultaneously. It is highly useful for demonstrating to the examiner that the code adapts to the hardware it runs on.

4. BFS & DFS Specific Logic Concepts
current_level (BFS): Replaces the standard queue's front. It holds all the nodes at the current depth of the graph that the threads are processing right now.

next_level (BFS): Replaces the standard queue's back. It is a globally shared vector where threads will dump all the unvisited neighbors they find. When the current_level is fully processed, it gets replaced by next_level.

local_next (BFS): A thread-local storage vector. Instead of all threads fighting to write to next_level at the same time (which requires heavy lock contention via critical), each thread gathers its own list of neighbors in local_next. Once finished, it dumps its entire local list into the global next_level at once, vastly speeding up the program.

dfsRecursive (DFS): A private helper function. Because OpenMP tasks require a function to call, we moved the actual depth-first search logic out of the main parallelDFS function and into this recursive helper.

-----------Part 1: Basic HPC and OpenMP TheoryExaminers usually start here to check your foundational knowledge.

1. What exactly is OpenMP?
OpenMP (Open Multi-Processing) is an Application Programming Interface (API) that supports multi-platform, shared-memory multiprocessing programming in C, C++, and Fortran. It is not a new programming language, but rather an extension added to existing languages.  

2. Is OpenMP used for distributed memory systems?
No, OpenMP is strictly designed for shared address space machines (shared memory). For distributed memory systems (like multiple separate computers linked together), we use MPI (Message Passing Interface).  

3. What is the fundamental difference between a Process and a Thread?
A process contains all the information needed to execute a program and has its own independent address space. A thread is the smallest unit of processing that an OS can schedule. Multiple threads can exist within the same process and share resources like global memory and address space, though each thread has its own run time stack and program counter.  

4. Explain the execution model of OpenMP.
OpenMP uses the Fork-Join model. The program begins as a single process called the master thread. When it encounters a parallel region, the master thread forks (creates) a team of threads. These threads execute the block concurrently. Once finished, they join (synchronize and terminate), leaving only the master thread to continue. 

 5. What is a "pragma" in C/C++?
"Pragma" stands for pragmatic information. It is a way to communicate instructions directly to the compiler. If a compiler doesn't understand the pragma (like compiling OpenMP code without the -fopenmp flag), it simply ignores it and produces a correct, sequential program.  

6. What are the major pros of using OpenMP?
It is easier to program and debug than MPI, allows for incremental parallelization (you can parallelize one loop at a time), and the same code can run sequentially if the OpenMP flags are ignored. 

 7. What are the cons of OpenMP?
It only works on shared-memory computers, requires a specific compiler that supports it, and there is a high chance of accidentally writing code with "false sharing" or race conditions if you aren't careful.  

----------------------Part 2: OpenMP Directives and SyntaxTesting your knowledge of the code you actually wrote.

8. What does #pragma omp parallel do?
It defines a parallel region. It tells the compiler to fork a team of threads to execute the block of code immediately following the directive.

9. What does #pragma omp for do?
It is a work-sharing construct. It takes the iterations of a standard for loop and divides them up equally among the team of threads created by #pragma omp parallel.

10. What is the use of the nowait clause in #pragma omp for nowait?
Normally, there is an implicit barrier at the end of a #pragma omp for loop where fast threads wait for slow threads to finish. nowait removes this barrier, allowing threads that finish their assigned loop iterations to immediately move on to the next line of code, reducing idle time.

11. What is a race condition?
A race condition occurs when multiple threads try to read and write to the same shared memory location at the exact same time, leading to unpredictable and corrupted data.

12. How did you prevent race conditions in your code?
By using #pragma omp critical. This directive creates a bottleneck ensuring that only one thread at a time can execute the block of code inside it, protecting shared variables like the visited array or cout.

13. What does #pragma omp single do?
It ensures that the enclosed block of code is executed by only one thread in the team (not necessarily the master thread, just whichever reaches it first). The other threads wait at the end of the single block unless a nowait is specified.

14. What does #pragma omp task do?
It defines an explicit task. Instead of executing immediately, the code block is packaged and placed into a queue. Any available thread in the parallel team can pick up the task and execute it. It is essential for irregular parallelism like recursion or while loops.

15. How do you measure execution time in OpenMP?
Using the omp_get_wtime() function, which returns the elapsed wall-clock time in seconds. We call it before and after the algorithm and subtract the difference.

16. How do you find out how many threads are executing your code?
By using omp_get_max_threads() to see the total hardware capability, or omp_get_num_threads() inside a parallel region to see how many threads are actively in the team.

17. What determines the number of threads OpenMP uses?
If not explicitly set by omp_set_num_threads() or the num_threads() clause, OpenMP defaults to the number of physical cores available on the hardware node.  

-----------------Part 3: Parallel Breadth-First Search (BFS)Defending your specific algorithmic choices.

18. How does standard BFS work?
It explores a graph level by level, radiating outward from the starting node. It completely explores all immediate neighbors before moving on to the neighbors' neighbors.

19. How did you parallelize BFS?
I used a level-synchronized approach.Instead of parallelizing the entire algorithm randomly, I parallelized the processing of nodes at the current depth level. All nodes in the current_level vector are processed simultaneously by different threads.

20. Why didn't you use a standard std::queue for Parallel BFS?
If multiple threads try to push and pop from a single std::queue, I would have to lock the entire queue using #pragma omp critical. This creates severe lock contention, forcing threads to wait in line, which ruins parallel performance.

21. So, what data structure did you use instead of a queue?
I used dynamic vectors (current_level and next_level) to represent the front and back of the queue. Threads read from current_level concurrently and push to next_level.


22. Explain the purpose of local_next in your BFS code.
local_next is a thread-local vector. Instead of every thread constantly fighting to lock the global next_level vector to add neighbors, each thread stores the neighbors it finds in its own private local_next vector. Once the thread finishes its loop, it locks the global vector just once to merge its local vector into it. This drastically reduces synchronization overhead.

23. Is the output of Parallel BFS always exactly the same as Sequential BFS?
The levels will always be printed in the correct order, but the order of nodes within the same level might shuffle. If Node 0 connects to Node 1 and Node 2, different threads process 1 and 2 simultaneously, so either could be printed first depending on thread scheduling.

24. What is the time complexity of Sequential BFS? 
O(V + E), where V is vertices and E is edges.

25. Does Parallel BFS change the theoretical time complexity?
No, the theoretical total work is still O(V + E). However, the wall-clock execution time is reduced because the work is divided among multiple processors.

26. What happens if a node has very few neighbors in BFS?
If the graph is very narrow (like a linked list), Parallel BFS actually performs worse than sequential BFS. Creating and joining threads has overhead; if there isn't enough work at a given level to distribute, the overhead outweighs the parallel benefits.


---------------------------Part 4: Parallel Depth-First Search (DFS)


27. How does standard DFS work?
It explores a graph by diving as deep as possible down one branch until it hits a dead end, then backtracks and explores the next available branch.

28. Why is DFS notoriously hard to parallelize?
Because its standard structure is inherently sequential. You have to finish exploring a child node before you can fully evaluate the parent's next step.

29. Why didn't you use std::stack for Parallel DFS?
Just like the queue in BFS, a shared standard stack accessed by multiple threads causes race conditions. If one thread pushes a node and another thread immediately pops it before the first thread is finished evaluating it, the graph traversal fails.

30. How did you parallelize DFS in your code?
I used OpenMP Tasks (#pragma omp task). Instead of maintaining an explicit data structure, I used recursion. Every time a thread finds an unvisited neighbor, it spawns a new task to recursively explore that neighbor.

31. Why use #pragma omp single at the start of your DFS?
To kick off the algorithm. If I didn't use single, every thread in the team would try to start a DFS traversal from the root node at the exact same time, doing redundant work. single ensures one thread starts the root task, which then spawns child tasks for the other threads to pick up.

32. In your dfsRecursive function, why do you check if (!visited[u]) inside a critical section?
To avoid a race condition. Two threads exploring different branches might find the same unvisited cross-edge node simultaneously. The critical section ensures only one thread "claims" the node and marks it visited, preventing both threads from traversing it.

33. Does Parallel DFS guarantee the exact same traversal order as Sequential DFS?
Absolutely not. Sequential DFS always visits the "leftmost" branch first. Parallel DFS dives down multiple branches simultaneously. The output will show nodes from entirely different branches interleaved as different threads print them.

34. Is Parallel DFS good for finding the shortest path?
No, BFS is used for shortest path in unweighted graphs. DFS is used for topological sorting, cycle detection, or exploring all possible paths.Part 5: Advanced Performance and HPC Concepts

35. If you have a 4-core machine, will your parallel code run exactly 4 times faster?
No, rarely. We almost never achieve perfect theoretical speedup due to several factors: Amdahl's Law (part of the code must remain serial), thread creation overhead, synchronization overhead (waiting at critical sections), and memory bandwidth limits.  

36. What is "Speedup"?
Speedup = (Execution time of sequential algorithm) / (Execution time of parallel algorithm).  

37. What is "Efficiency"?
Efficiency = Speedup / Number of processes or threads used. It measures how well the processors are being utilized.  

38. What is "False Sharing"?
False sharing occurs when two or more threads independently modify variables that happen to reside on the exact same cache line in the CPU. Even though they aren't sharing the exact variable, the CPU has to constantly invalidate and fetch the cache line, severely degrading performance. 

39. How do we avoid False Sharing?
By padding variables or using thread-local storage (like the local_next vector in our BFS code), ensuring threads work on distinct chunks of memory before merging results.

40. What is "Load Balancing"?
It is the process of distributing work equally among threads. If one thread gets a massive branch of the graph to explore and the other threads get small branches, the whole program runs as slow as the longest thread. #pragma omp task helps with dynamic load balancing because free threads automatically grab pending tasks.  

41. What is an Adjacency List and why did you use it over an Adjacency Matrix?
An adjacency list represents a graph as an array of vectors. It is highly space-efficient for sparse graphs ($O(V+E)$ space). An adjacency matrix requires $O(V^2)$ space, which wastes massive amounts of memory for large graphs with few connections.

42. How does your code handle an isolated node (a node with no edges)?
Because we dynamically allocate the vectors based on the number of nodes V, an isolated node simply has an empty neighbor list in adj. The loop for (int v : adj[u]) will immediately terminate for that node, gracefully handling the edge case.

43. Is a Tree an undirected graph?
Yes, a tree is a special type of undirected graph that is fully connected and contains absolutely no cycles.

44. If I give your code a Tree input, does it require any modification?
No modifications are required. Since the code safely marks nodes as visited and checks edges bidirectionally, it will traverse a tree perfectly without getting stuck in infinite loops.

45. What happens if there is a cycle in the graph?
The visited[v] boolean check saves the algorithm. When a thread encounters a node that is already part of the cycle, it sees visited == true and simply skips it, breaking the loop.

46. Can OpenMP parallelize a simple while loop directly?
No. OpenMP's #pragma omp for requires a loop with a knowable, finite number of iterations at the start (like a standard for loop). To parallelize while loops (like in BFS), we must wrap a parallel for loop inside the while loop, or use OpenMP Tasks.

47. Explain the concept of "Thread Contention" or "Lock Contention".
It happens when multiple threads frequently try to acquire the same lock (like #pragma omp critical). While one thread holds the lock, all other threads must sit idle, wasting CPU cycles and severely degrading parallel performance.

48. Why is measuring performance in milliseconds/seconds tricky for small graphs?
For very small graphs (like 5 or 10 nodes), the overhead time it takes the OS to create and destroy the OpenMP threads is actually longer than the time it takes to just traverse the graph sequentially. To truly see the benefits of parallelization, you must test on massive graphs with thousands of nodes.

49. What does the -fopenmp compiler flag physically do?
It links the OpenMP runtime library (libgomp in GCC) to your program and tells the compiler to parse and execute the #pragma omp directives. Without it, the compiler treats the pragmas as comments.
