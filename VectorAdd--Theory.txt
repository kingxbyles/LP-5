Part 2: The CUDA Kernel (The GPU Function)A "Kernel" is simply a C++ function that is called by the CPU (Host), but executed simultaneously by thousands of threads on the GPU (Device).

C++

__global__ void addVectors(int* A, int* B, int* C, int n) {

__global__: This is a CUDA-specific keyword. It tells the compiler: "This function is a Kernel. It will be launched by the CPU, but it will run on the GPU."int* A, int* B, int* C: These are pointers to the arrays that exist in the GPU's memory.

C++    

int i = blockIdx.x * blockDim.x + threadIdx.x; 


The "Magic Formula": This is the most famous line in CUDA programming. It calculates the Global Thread ID. Since thousands of threads run this exact same code at the same time, every thread needs a unique ID to know which element of the array it should add.blockDim.x: The number of threads in one block (we set this to 256 later).blockIdx.x: Which block this thread belongs to (e.g., Block 0, Block 1, Block 2).threadIdx.x: The thread's local ID inside its specific block (from 0 to 255).Example: If a thread is ID 5 (threadIdx.x) inside Block 2 (blockIdx.x), its global ID is: 2 * 256 + 5 = 517. This thread will strictly add the 517th element of the arrays!

C++    
if (i < n) {
        C[i] = A[i] + B[i];
    }
}

if (i < n): This is a Boundary Check. If our array has 10 elements, but the GPU launches a block of 256 threads, threads 10 through 255 will try to do math on memory that doesn't exist. This if statement ensures only the threads within the array size actually perform the addition.

size_t size = n * sizeof(int);: Arrays need a specific amount of bytes in memory. An integer (int) usually takes 4 bytes. If we have 1,000,000 elements, we need 4,000,000 bytes. This line dynamically calculates exactly how much memory space is required.

Part 4: GPU Memory Allocation & Data Transfer

C++    

cudaMalloc(&d_A, size);

    cudaMalloc(&d_C, size);


cudaMalloc: Stands for "CUDA Memory Allocation." It asks the GPU to carve out empty space in its own dedicated VRAM (Video RAM) to hold our arrays. We pass the memory address & and the size in bytes.

C++   

 cudaMemcpy(d_A, h_A, size, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, size, cudaMemcpyHostToDevice);

cudaMemcpy: Stands for "CUDA Memory Copy."cudaMemcpyHostToDevice: This is a specific directional flag. It tells the computer to physically copy the data from the normal CPU RAM (h_A) over the motherboard's PCIe lane directly into the GPU's VRAM (d_A).Part 5: Execution Config & Kernel Launch

C++   

 int threadsPerBlock = 256;
    int blocksPerGrid = (n + threadsPerBlock - 1) / threadsPerBlock; 

threadsPerBlock = 256: NVIDIA GPUs group threads into "Blocks". 256 is the standard, highly optimized number of threads per block.blocksPerGrid calculation: This is a mathematical trick for Ceiling Division. If we have 1,000,000 elements and 256 threads per block, $1000000 / 256 = 3906.25$. Since we can't have a fraction of a block, this formula forces it to round up to 3907 blocks, ensuring every single element gets processed.

C++    
addVectors<<<blocksPerGrid, threadsPerBlock>>>(d_A, d_B, d_C, n);


<<< ... >>>: This is the Execution Configuration Syntax. It tells the compiler how to launch the GPU Kernel. It dictates the exact grid dimensions (how many blocks to use, and how many threads inside each block).Notice we pass d_A, d_B, d_C. We must pass the Device pointers, because this function is running on the GPU!

C++    cudaDeviceSynchronize(); 


cudaDeviceSynchronize(): When the CPU launches a Kernel, it doesn't wait for the GPU to finish; it instantly moves to the next line of code. If the CPU tries to copy the results back immediately, it will grab empty data because the GPU is still doing the math! This keyword forces the CPU to "pause and wait" until all GPU threads have successfully finished their work.

Part 6: Bringing Data Back and Cleanup

C++    

cudaMemcpy(h_C, d_C, size, cudaMemcpyDeviceToHost);


cudaMemcpyDeviceToHost: The GPU has finished the addition and stored the answers in d_C. We must now copy those answers backward across the motherboard into the CPU's memory (h_C) so we can actually print and use them.(Note: The verification for loop in the code simply loops through the CPU arrays and checks if $A + B = C$ to print a success message).

cudaFree(d_A);

    delete[] h_A;
    delete[] h_B;
    delete[] h_C;

}

cudaFree(): This is incredibly important. It deletes the arrays from the GPU's VRAM. Part 1: CUDA & GPU Basics

1. What is CUDA?
CUDA (Compute Unified Device Architecture) is a parallel computing platform and programming model created by NVIDIA. It allows developers to use the GPU for general-purpose processing (GPGPU), not just graphics.



2. What is the difference between CPU and GPU?
•	CPU: Has a few highly powerful cores optimized for sequential serial processing and complex logic.
•	GPU: Has thousands of smaller, simpler cores designed for massive parallel processing (handling multiple simple tasks at the exact same time).

3. What do the terms "Host" and "Device" mean in CUDA?
•	Host: Refers to the CPU and the system's main RAM.
•	Device: Refers to the NVIDIA GPU and its dedicated Video RAM (VRAM).

4. Can CUDA run on Intel or AMD graphics cards?
No. CUDA is proprietary and strictly works only on NVIDIA GPUs. (OpenCL or HIP would be used for AMD/Intel).


5. What is a "Kernel" in CUDA?
A Kernel is a C++ function that is called by the Host (CPU) but executed simultaneously by thousands of threads on the Device (GPU).

6. What is the file extension for a CUDA program?
.cu (e.g., vector_add.cu).

7. What compiler is used to compile CUDA code?
nvcc (NVIDIA CUDA Compiler).
________________________________________

Part 2: CUDA Memory Management


8. How do you allocate memory on the Host (CPU)?
Using standard C++ functions like new, malloc(), or calloc().


9. How do you allocate memory on the Device (GPU)?
Using cudaMalloc(). It takes the address of the pointer and the size of the memory required in bytes.

10. How do you transfer data between the CPU and GPU?
Using the cudaMemcpy() function.

11. What are the direction flags used in cudaMemcpy?
•	cudaMemcpyHostToDevice (CPU to GPU)
•	cudaMemcpyDeviceToHost (GPU to CPU)
•	cudaMemcpyDeviceToDevice (GPU to GPU)

12. Why is transferring data between Host and Device considered a "bottleneck"?
Because the data has to physically travel across the motherboard's PCIe bus, which is much slower than the internal speed of the GPU or CPU.

13. Why must we use cudaFree() at the end of the program?
To prevent Memory Leaks. If we don't free the device memory, the GPU will permanently hold onto that space until the system restarts, eventually causing the GPU to run out of memory.

14. What happens if you try to read a Device pointer (d_A) directly on the Host (CPU)?
The program will crash (Segmentation Fault). The CPU cannot directly access GPU memory; it must be copied back using cudaMemcpy first.

________________________________________

Part 3: The CUDA Thread Hierarchy

15. How does CUDA organize its threads?
CUDA organizes threads into a two-level hierarchy: Blocks and Grids.
•	A group of threads makes a Block.
•	A group of blocks makes a Grid.

16. Why use both Blocks and Grids instead of just a massive list of threads?
For Scalability. Blocks can be scheduled on different multi-processors independently. This allows the exact same compiled code to run on a cheap laptop GPU (running a few blocks at a time) or a massive supercomputer GPU (running thousands of blocks at a time).

17. What is threadIdx.x?
It is a built-in CUDA variable that gives the unique ID of the thread inside its specific block (e.g., thread 0 to 255).

18. What is blockIdx.x?
It gives the unique ID of the block inside the grid (e.g., Block 0, Block 1...).

19. What is blockDim.x?
It gives the total number of threads inside a single block (in our code, we set this to 256).

20. Explain the "Magic Formula": int i = blockIdx.x * blockDim.x + threadIdx.x;
This calculates the Global Thread ID across the entire grid.
Example: If a block has 256 threads (blockDim.x), and you are looking at Thread 5 (threadIdx.x) inside Block 2 (blockIdx.x), its global index is $(2 \times 256) + 5 = 517$. This thread will process the 517th element of the vector.

21. What is the maximum number of threads you can put in a single block?
On modern GPUs, the limit is 1024 threads per block.

22. What is a "Warp" in CUDA?
A Warp is a group of 32 threads that execute the exact same instruction at the exact same time. The GPU schedules work in warps, not individual threads.

Part 4: The Vector Addition Kernel


23. What does the __global__ keyword mean?
It signifies that the function runs on the Device (GPU) but is called from the Host (CPU).

24. What does the __device__ keyword mean?
It signifies a function that runs on the GPU and can only be called by other functions already on the GPU.

25. Can a __global__ function return an integer or float?
No. Kernel functions must always have a void return type. Results must be written directly into device memory pointers.

26. Why do we need the Boundary Check: if (i < n)?
If we have 1,000,000 elements, and our block size is 256, the GPU will launch exactly 3,907 blocks (999,944 to 1,000,192 threads). The boundary check ensures that threads 1,000,000 through 1,000,191 do not try to access memory that doesn't exist, which would crash the program.

27. What is the Execution Configuration syntax?
The <<<blocksPerGrid, threadsPerBlock>>> syntax. It tells the compiler the exact dimensions of the grid to launch for the kernel.

28. Why do we use cudaDeviceSynchronize()?
Kernel launches are Asynchronous. The CPU launches the kernel and immediately moves to the next line of code without waiting. cudaDeviceSynchronize() forces the CPU to pause and wait until the GPU has 100% finished its math before copying the answers back.

________________________________________

Part 5: Time / Space Complexity & Performance


29. What is the Time Complexity (TC) of Sequential CPU Vector Addition?
$O(N)$. A single CPU core must iterate through a for loop $N$ times, adding one pair of numbers at a time.

30. What is the Time Complexity (TC) of Parallel GPU Vector Addition?
$O(N / P)$, where $P$ is the number of threads. If we have enough threads, the time complexity approaches $O(1)$ mathematically, because every single pair is added simultaneously in a single step!

31. What is the Space Complexity (SC) of the program?
$O(N)$. We require arrays of size $N$ on both the Host RAM and the Device VRAM.

32. What is "Compute-Bound" vs "Memory-Bound"?
•	Compute-Bound: The program is limited by how fast the processor can do math (e.g., calculating Pi).
•	Memory-Bound: The program is limited by how fast data can be fetched from RAM.

33. Is Vector Addition Compute-Bound or Memory-Bound?
It is highly Memory-Bound. Adding two numbers together takes a fraction of a nanosecond, but moving 1,000,000 elements from CPU to GPU, and fetching them from VRAM, takes much longer.

34. If GPU is so fast, why don't we use it for an array of 10 elements?
Because the time it takes to allocate GPU memory and copy the data over the PCIe bus (Transfer Overhead) is 100x slower than just letting the CPU add the 10 numbers itself. GPUs are only beneficial for massive datasets.

Part 6: Scenario-Based "Trick" Examiner Questions


35. Examiner: "What happens if I forget to copy the result back using cudaMemcpyDeviceToHost?"
Answer: "The GPU will do the math perfectly, but the CPU's array (h_C) will just contain empty garbage values. When you print the output, it will be entirely wrong."

36. Examiner: "Explain this formula: (n + threadsPerBlock - 1) / threadsPerBlock"
Answer: "This is a standard integer math trick for Ceiling Division. It ensures that if $N$ is not perfectly divisible by the block size, it adds one extra block to cover the remaining elements. (e.g., $10 / 3 = 3$, but $(10 + 3 - 1) / 3 = 4$ blocks)."

37. Examiner: "Why did you choose 256 threads per block?"
Answer: "256 is a multiple of 32 (a Warp size). Keeping thread counts as multiples of 32 ensures 100% hardware utilization without leaving cores idle."

38. Examiner: "Can Thread 5 in Block 1 directly share a variable with Thread 5 in Block 2?"
Answer: "No. Threads in different blocks cannot easily communicate or synchronize with each other. They can only communicate by writing to the global device memory."

39. Examiner: "What happens if you remove cudaDeviceSynchronize() but keep cudaMemcpy right after the kernel call?"
Answer: "Surprisingly, it will usually still work! cudaMemcpy from Device to Host acts as an implicit synchronization barrier. It will naturally wait for the GPU to finish before copying. However, it is a best practice to include explicit synchronization to catch errors."

40. Examiner: "Can I use CUDA to add two strings together?"
Answer: "Technically yes, if you convert characters to ASCII integer arrays, but GPUs are strictly designed for high-throughput floating-point and integer mathematics, not complex string manipulation."

41. Examiner: "What is nvcc and how does it work?"
Answer: "nvcc is a compiler driver. It separates the code: it sends the standard C++ CPU code to the native host compiler (like g++), and it compiles the GPU code (__global__) into PTX assembly code for the NVIDIA graphics card."

42. Examiner: "What does the %%cu command do in Google Colab?"
Answer: "It is a Jupyter Notebook 'cell magic' command provided by the nvcc4jupyter plugin. It tells the cell to write the contents to a temporary .cu file, compile it with nvcc, and execute it, all in one click."

43. Examiner: "If I have an array of 3 million elements, will the GPU physically spawn 3 million physical cores?"
Answer: "No. A GPU might only have 4,000 physical cores. CUDA uses Context Switching. It maps the 3 million software threads to the 4,000 physical cores. When one thread is waiting for memory, the hardware instantly swaps in another thread to do math."

44. Examiner: "What is the difference between OpenMP (Practical 2 & 3) and CUDA (Practical 4)?"
Answer: "OpenMP is for multi-core CPUs (e.g., 8 to 16 cores) sharing the same motherboard RAM. CUDA is specifically for NVIDIA GPUs (thousands of cores) which have their own isolated Video RAM."

45. Examiner: "Is it possible to use 1D, 2D, and 3D grids?"
Answer: "Yes. We used blockIdx.x (1D) because vector addition is a flat 1D array. If we were doing Matrix Multiplication or Image Processing, we would use blockIdx.x and blockIdx.y to map threads to a 2D grid."

46. Examiner: "What would happen if your boundary check if(i < n) was removed?"
Answer: "The extra threads at the end of the final block would calculate an index that goes past the end of the array. It would overwrite random GPU memory, potentially corrupting other programs or crashing the GPU."

47. Examiner: "Can we initialize the arrays directly on the GPU without using the CPU?"
Answer: "Yes! You can write a separate simple Kernel function that runs on the GPU and assigns A[i] = i. This avoids the slow Host-to-Device memory copy entirely."

48. Examiner: "What does size_t mean?"
Answer: "size_t is an unsigned integer type used to represent the size of objects in bytes. It is guaranteed to be large enough to contain the size of the biggest object the system can handle, making it safe for massive arrays."

49. Examiner: "Why is CUDA so important in today's tech landscape?"
Answer: "Because Artificial Intelligence and Deep Learning (like ChatGPT) rely fundamentally on multiplying millions of matrices and vectors together. CPUs are too slow for this; CUDA allows GPUs to train neural networks in days instead of years."

50. Examiner: "Summarize the 5 main steps of any CUDA program."
Answer: 1. Allocate CPU and GPU memory.
2. Copy data from CPU to GPU.
3. Launch the Kernel (Execution Configuration).
4. Copy the results from GPU back to CPU.
5. Free all memory.
